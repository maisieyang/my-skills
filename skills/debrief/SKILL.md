# /debrief — 模块收工回顾：把实现变成认知

> ⚠️ **draft**：未经实测打磨。

写完代码就过去，是执行；把实现里的判断挖出来探讨过，才是学习。本 skill 防的是"代码进了仓库，认知没进脑子"——AI 写得越快，这个缺口越大。

**终点信号：`docs/modules/<module>.md` §回顾 写完——这个模块在领域专家书里成为完整一章（设计立场 + §回顾）。**

## 流程

1. **证据收集**（AI 准备，human 不翻 git log）：模块起点以来的 commits 概览 / diff 规模 / 测试数量与覆盖 / 与设计阶段（interview-me/plan）核心要素清单的对照——哪些做了、哪些没做、哪些做了但没经过设计探讨。
2. **隐式决策挖掘**——核心环节，必须是对话：从 diff 里找出**实现过程中没经过探讨就定了的设计决策**（数据结构选型、错误处理策略、边界行为、某个抽象的命名……）。逐个摆出来探讨：当时为什么这么选？有什么别的选法？现在还会这么选吗？
   - 这是对"执行期 review 不过来"的补课：当时快速通过的判断，在这里以模块为单位、以可负担的带宽补回探讨。
3. **设计证伪**：设计阶段（interview-me/plan）定的判断/立场，哪些被实现推翻了？trade-off 有没有在实现压力下偷偷换边？预测错误暴露的认知偏差写下来——下个模块设计时还会再犯，除非这里写清。**没有设计基线**（老模块）就对照 REFERENCE §3 + diff 自评，§回顾 标注"回溯重建"。
4. **行业对比回看**：建之前看对标项目和代表实现，是"知道他们怎么做"；建完自己的版本再看，是"理解他们为什么这么做"。重读参照实现（对象见项目 CLAUDE.md 绑定）：哪些当初看不懂的选择现在看懂了？哪些我们的做法有理由相信更好？
5. **写 §回顾**进 `docs/modules/<module>.md`：实现概要（它实际怎么工作，写给未来的自己和外部读者）/ 关键决策与 trade-off（含隐式决策补录）/ 被证伪的设计判断 / 行业对比的新理解 / open questions + 对后续模块的预判（留待下一轮回顾证伪）。
6. **质量收尾**：对模块 diff 跑一次内置 `/code-review` 深度走查；发现的问题要么修掉、要么显式记进 open questions——不许默默放过。

## 容器纪律

- 模块文档是领域专家书的一章，narrative 欢迎——写给未来的自己和外部读者
- **不自动写入 AI 持久 memory**——哪些认知值得进 memory 由 human 之后自己判断（显式决策，不是遗漏）
- 使用中发现的概率行为失败 → 固化成永久回归用例，不能只记在回顾里（只记不固化，等于没防住它再次发生）

## 不要做

- 不写成就清单——主角是判断与误差，不是完成的功能列表
- 不在没有探讨的情况下单方面写完 §回顾——隐式决策挖掘环节跳过了，这次 debrief 就白做了
- 不重写历史——设计阶段被证伪的立场保持原样，§回顾 里记 correction（决策历史可追溯 > 文档好看）
