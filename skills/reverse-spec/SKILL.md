# /reverse-spec — 逆向对标：把行业水平变成你的认知地图

新手最难的不是写代码，是建立一个领域的知识体系。没有参照系直接 build-my-own-XX，极易做成玩具——你不知道核心部分有哪些、专业实现分几层、哪些复杂度是本质的。

本 skill 逆向一个优秀的对标项目，把它的解剖图变成你的认知起点。

**产出定位**：REFERENCE.md 是**认知脚手架，不是工程合同**。它回答"这个领域的专业系统由哪些核心要素构成、每个解决什么问题"。它不要求你 follow 它的实现——具体怎么实现由你和 AI 重新创作。

**终点信号**：一个新手读完，能做到三件事（对应三个镜头）——
1. 看**目录树 + 分层**，说出这个专业系统**怎么组织、怎么分层**（企业级应用的结构长什么样、为什么这么切）；
2. 看**概念地图 §3**，答出它**由哪些核心要素构成、每个解决什么问题、各有什么取舍**；
3. 看**数据流**，讲出它**怎么 run 起来**（一条输入从进到出怎么流）。

三件齐了，认知才算立住——只答得出 ② 还差两条腿。

## 选对标的三个标准

1. **对标行业水平**——它本身就是或对标行业标杆（例：harness 领域选 OpenHarness，因为它对标 Claude Code）。
2. **你有用户视角的体感**——你深度用过它或它的对标物。
3. **活跃的开源项目**——跟还在进化的优秀项目学，不跟尸体学。

## 流程

> **三个镜头,两种形态**：目录树（结构图）+ 数据流（流程图）是**视觉镜头**（用图，看得到结构和流动）；§3 概念地图是**认知镜头**（用文字，讲 why）。§3.④ 是概念地图 ↔ 目录树之间的桥。

1. **入口考古 + 目录树镜头**：README / 入口点 / 技术栈 / 一句话定位；并**如实捕获带注解的原始目录树**（目录级，每行一句话职责，照上游结构、不分组、不抽象）。**用树形图呈现**（ASCII 树，看得到层级）——这是"专业团队怎么把系统切成目录"的写照，实现层的宏观架构，第一次读源码一次性读出。
2. **架构骨架 + 数据流**：分层 + "一条用户输入从进到出经过谁"。**两者都用图呈现**（栈式框图 + boxes-and-arrows 流程图）——**真的能看到层级结构和数据流动**，别用流水账列表。**这一节是 §3 的来源**——读完它，才问得出"这个系统必须解决哪几个本质问题"。
3. **核心要素概念地图**：从 §2 提炼这个领域的核心要素，每个一章。**不照上游目录走**（那是代码地图）。判定见下方两个测试。
4. **横切两章**：① 设计决策 / 跨要素反复出现的模式；② 与你熟悉产品的对比。§3 按要素纵切，§4 横跨要素看。
5. **头部声明**：source attribution + 上游 license + **版本锚点** + "独立重建、无代码拷贝"。

§1-§4 + 头部声明是 REFERENCE 的冻结认知层（不做实现精度附录——常量/算法精度等真建到那块再读源码）。

6. **§5 模块拆分**（把 §3 切成 build 模块，作为 REFERENCE 最后一章——和 §1-§4 一样**冻结**）：
   - **模块 = 一次设计 + 一个 build arc + 一篇回顾**。原子要素 1:1 一个模块；簇型要素（一个 §3 概念含多个能独立 build 的子能力）按 §3.④ 的子目录缝拆开。
   - **列表顺序 = 依赖拓扑序 = 建造顺序**：先建能让一条输入端到端跑通的最小骨架，再按依赖往外加层。每个模块标"依赖谁 + 一句拆分/排序理由"。
   - **只拆 + 排序，不追踪进度**：§5 是结构（谁依赖谁、谁先谁后），不是进度板。状态 / 谁 done / 选下一个 **不进 REFERENCE**——去 `tasks/plan.md`、git、你脑子。

## 两个判定测试

**测试一：这是"核心要素"还是"renamed 的文件夹"？**

> 删掉上游所有目录名，你还能从 §2 数据流推出它吗？

- 能 = 核心要素。本质要素 = **每条用户输入都必经的关口** + 系统为了不死必须解决的硬问题。
- 只是目录名 = 不是要素。
- **一个要素常横跨好几个上游模块**——例：OpenHarness 的"安全边界" = permissions + hooks + sandbox 三个目录合起来。如果你的要素和上游目录 1:1 对上，多半是没提炼、照搬了目录。
- **完整性**对照 §2 数据流（每条路径、每个硬问题都被覆盖），**不**对照目录列表。
- 注：这条说的是 **§3 的组织轴不能是目录**。**如实捕获目录树是另一回事**（见 §1 目录树镜头）——树照实存，§3 按问题重组，两者并置、§3.④ 连接。

**测试二：这条该写进认知（原则），还是该丢掉（实现精度）？**

> 换一个实现，这条还成立吗？

- 成立 = **原则，写**。例："压缩绝不能切断 tool_use/tool_result 配对"（任何压缩实现都得守）。
- 会变 = **常量/实现，不写**。例：`BUFFER=13000`、触发阈值的具体数字、`split_index` 怎么回退（都是这个实现的调参）。

## 每个要素章写什么（四段）

1. **解决什么问题**（why，不是 what）。
2. **关键 trade-off**——上游怎么权衡的。
3. **最小接口形状**——1-5 行，只到"对外长什么样"，方法体/字段细节不写。
4. **上游在哪几个目录/模块实现**——指针，**也是概念地图 ↔ 目录树（§1 那棵）之间的桥**：每个要素 → 树上哪几个节点。锚定头部版本，上游移了重新定位。

## 读源码的尺度（别因为"lazy"就编）

**你现在就要读源码**，读到能诚实写出上面四段的程度：看出每个要素的对外 shape、上游的 trade-off、定位到实现模块。"lazy 读源码"**只适用于重建级的常量/算法精度**——那些等你真建到那块再读。

trade-off 常常不在注释里。可以**从代码形态推断**，但标注"(推断)"和"(上游明说)"。真看不出就写"未见上游说明"——**不要编**一个似是而非的 trade-off 来填表。这是最容易产出"通过检查表但全是空话"文档的地方。

## 纪律

- **按要素发现，不按目录照搬**（最重要的一条）——要素从 §2 数据流提炼，不是 src 目录列表的复刻。
- **现在就读源码**到"懂 shape + trade-off + 能定位"；lazy 只管重建级精度。
- **写"他们怎么做"，不写"我会怎么做"**——逆向忠实记录上游立场，改良意见是后续设计的事。
- **不抄代码**——最小接口形状用伪代码 + 表格。
- **源码没有的不硬写**——trade-off 可推断但标注，没有就说没有，别编。
- **产出要可读**：并列项 >3 用列表/表格，不用中点长串；流程一步一行，分支显式展开；每个要素章 ~10-15 行；整份是"读得完的地图"，不是"查得到的手册"。

## domain-generic：每个领域自己发现要素

这 skill 逆向**任何**领域，不只 harness。要素清单不能跨领域套用，必须从那个领域的数据流重新发现。示例（非 harness，感受形态怎么迁移）：

- 逆向**数据库引擎** → 查询解析 / 执行计划 / 存储引擎 / 事务 / 并发控制
- 逆向 **web 框架** → 路由 / 中间件 / 请求生命周期 / ORM / 模板渲染

harness 的"主循环/上下文/工具/权限/记忆/多Agent/扩展"只是 harness 这个领域的答案，不是模板。

## Common Rationalizations

| 借口 | 现实 |
|---|---|
| "照着上游目录逐模块写，省事又全" | 那是代码地图——跳过"发现核心要素"这步认知工作。过测试一：删掉目录名还推得出吗？ |
| "lazy 读源码，先不读，trade-off 先写着" | 那会编出空话。现在就读到懂 shape + trade-off；lazy 只管重建级常量。 |
| "这条算原则还是常量，拿不准就都写上" | 过测试二：换个实现还成立吗？会变的就是这个实现的调参，丢。 |
| "某个要素不重要，跳过" | 对照 §2 数据流查完整性——漏的那块是你将来玩具化的起点。 |
| "上游设计得不好，我按我的想法写" | 参照系必须忠实于参照物。改良是后续设计的事。 |
| "等上游下个版本再发布" | 冻结发布。版本锚点 + 指针过期时重新定位即可。 |
| "有了目录树就照它组织 §3" | 树是"如实快照"的镜头（他们怎么码的），§3 是"按问题重组"的认知。两者并置 + §3.④ 连接——别拿树当 §3 的组织轴，那又回代码地图了。 |
